## Current Real Implementation

- `server` is the only Spring Boot business backend.
- `mock-carrier` is a local simulator that sends webhook payloads for two different carrier formats.
- Webhook HMAC verification exists and uses constant-time comparison.
- The system stores raw carrier events, normalized events, event process tasks, anomalies, reconciliation batches/tasks, operation logs, shipments, carriers, and carrier status mappings.
- Shipment state replay sorts normalized events by `event_time`, `received_time`, and `id`.
- UNKNOWN and invalid transitions are stored and can create anomalies.
- The Vue console displays dashboard, shipments, raw events, anomalies, reconciliation batches, carriers, and simulation.

## README Claims Not Fully Implemented

- README mentions RabbitMQ in deployment, but `server` has no Spring AMQP dependency, queue, exchange, producer, consumer, manual ACK, publisher confirm, or DLQ code.
- README describes active reconciliation, but current `CoreService.reconcile()` inserts a fixed status tail instead of querying the carrier.
- README says Docker Compose contains RabbitMQ, but the application does not use it yet.
- E2E tests exist for the frontend, but CI currently does not run Playwright E2E.

## Confirmed Code Problems

- `CoreService.receiveWebhook()` calls `processRawEvent(rawId)` in the webhook HTTP transaction, so webhook handling is not a quick asynchronous accept flow.
- `event_process_task.retry_count`, `next_retry_time`, `locked_at`, and `locked_by` are not used as a real retry/claim mechanism.
- The scheduled recovery directly calls `processRawEvent()` and does not implement finite retry semantics.
- `CoreService` currently owns webhook intake, event processing, state replay, anomaly handling, dashboard queries, reconciliation, and scheduled recovery.
- `@RequestBody` parameters in controllers are not annotated with `@Valid`.
- Anomaly status is accepted as arbitrary `String`.
- `shipment.version` is incremented, but no optimistic-lock `WHERE id=? AND version=?` update exists.
- `normalized_event` has a global unique constraint on `event_fingerprint`; it should be unique per carrier.
- Event fingerprint generation does not include `carrierId` or `carrierCode`.
- `request_headers` avoids saving the full signature, which is good, but error responses are still framework-default.
- `mock-carrier` has no complete trajectory query endpoint.
- `MISSING_EVENT` removes an event from the only generated list, so the simulator cannot later return a complete trajectory for reconciliation.

## Refactoring Plan

1. Phase 1: align Flyway schema with MySQL-compatible syntax, add Maven Wrapper, add request validation, add typed error responses, keep baseline tests passing.
2. Phase 2: add Spring AMQP, RabbitMQ topology, Outbox table and publisher, manual-ACK consumer, task claiming, finite retries, and DLQ behavior.
3. Phase 3: split event processing and shipment replay responsibilities, fix carrier-aware fingerprints, make replay concurrency-safe.
4. Phase 4: add complete mock carrier trajectories, query client, and real reconciliation compensation.
5. Phase 5: expose task/retry/reconciliation details in frontend, update Docker/CI/docs/interview materials.

## Features Not Being Added

- No Spring Cloud, Nacos, Eureka, Gateway, Feign, Seata, Dubbo, Kubernetes, service registry, distributed transaction framework, Redis, JWT, Elasticsearch, Kafka, or complex RBAC.
- No microservice split. `server` remains the only business backend.
- No Exactly Once claim. The intended messaging model is at-least-once delivery plus idempotent database writes.

## Phase 2 Changes Completed

- Added `OutboxPublisher`:
  - scans `PENDING` and due `RETRY_WAIT` outbox rows;

- Added `EventProcessConsumer`:
  - consumes from `trackflow.event.process.queue`;
  - calls database task claiming before processing;
  - uses manual ACK;
  - rejects terminal failed messages without requeue so they can reach the DLQ.

- Changed webhook intake so it writes raw event, event task, and outbox event in the same transaction and no longer calls full event processing in the HTTP thread.
- Added event task query and manual retry APIs.
- Changed scheduled recovery so it re-enqueues due tasks through Outbox instead of directly processing raw events.
- Added `.env.example` with demonstration-only values.
- Added Docker Compose RabbitMQ-related environment variables for `server`.
- Added `WebhookOutboxTest`, proving legal webhook intake creates raw/task/outbox records and does not synchronously create normalized events; duplicate webhook does not create duplicate task/outbox rows.
