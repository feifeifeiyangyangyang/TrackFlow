# 10 分钟演示流程

## 1. 打开控制台

访问 `http://127.0.0.1:8001`，先看“数据概览”。说明这个平台不是普通物流查询页，而是治理多物流商履约事件的一套后台。

## 2. 模拟物流商推送

进入“故障模拟”，分别执行：

- `NORMAL`：正常轨迹。
- `DUPLICATE_PUSH`：重复推送。
- `OUT_OF_ORDER`：乱序到达。
- `MISSING_EVENT`：漏推事件，后续用主动对账补偿。

## 3. 查看运单详情

进入“运单管理”，打开运单详情页，讲三个时间概念：

- `eventTime`：物流事件真实发生时间。
- `receivedTime`：系统收到 Webhook 的时间。
- `lateArrival`：后收到但业务发生时间更早的迟到事件。

## 4. 查看任务链路

进入“处理任务”，展示事件处理任务的状态、重试次数、执行者和错误信息。这里可以讲 Outbox、RabbitMQ、消费者抢占和手动 ACK。

## 5. 主动对账

对 `MISSING_EVENT` 运单发起对账，然后进入“对账任务”查看 batch 和 task。对账消费者会查询 Mock Carrier 完整轨迹，只补入缺失事件。

## 6. 面试收束话术

可以这样总结：

> TrackFlow 关注的不是单次接口成功，而是事件在重复、乱序、漏推和异常情况下仍然可追踪、可重试、可补偿。核心设计是原始事件留存、幂等写入、Outbox 可靠消息、任务状态机和按业务时间重放。
