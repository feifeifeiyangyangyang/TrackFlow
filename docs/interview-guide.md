# TrackFlow 面试讲解指南

## 30 秒项目介绍

TrackFlow 是一个多物流商履约事件治理平台。它接收不同物流商的 Webhook，解决重复推送、乱序到达、漏推和状态冲突问题。系统会保存原始事件，转换成统一事件，再按业务发生时间重放状态机，最终得到可信的运单状态，并通过异常中心和主动对账修复问题。

## 2 分钟技术介绍

这个项目采用前后端分离架构：

- 前端使用 Vue 3 + TypeScript + Element Plus，提供运营后台、运单时间线、异常中心和故障模拟页面。
- 后端使用 Spring Boot 3，核心模块包括 Webhook 验签、物流商适配器、幂等保存、状态机重放、异常检测和主动对账。
- 数据层通过 Flyway 初始化表结构，用 `raw_carrier_event` 保留原始请求，用 `normalized_event` 存标准化事件，用 `shipment` 保存当前可信状态。
- Mock Carrier 模拟两个物流商，故意设计成字段名、时间格式、状态码都不同，用来证明适配器模式和统一状态模型有效。

## 最值得讲的 5 个设计点

### 1. 为什么要区分 eventTime 和 receivedTime？

物流事件真实发生时间和系统收到时间不一定一致。例如“已签收”可能先到，“运输中”可能后到。如果只按接收顺序更新状态，就会导致签收后的运单被回退成运输中。

TrackFlow 的做法是：

1. `eventTime` 表示物流业务真实发生时间。
2. `receivedTime` 表示系统收到 Webhook 的时间。
3. 状态重建时按 `eventTime -> receivedTime -> id` 排序。

这个点适合回答“你怎么处理乱序消息？”

### 2. 幂等为什么要靠数据库唯一约束兜底？

Webhook 可能被物流商重复推送，也可能因为网络重试并发打进来。单纯用 Java 内存 Set 不可靠，服务重启会丢，并发下也可能漏。

TrackFlow 的做法是：

- 有外部事件号时使用 `EXT:{externalEventId}`。
- 没有外部事件号时，用运单号、状态、时间、地点、描述生成 SHA-256 指纹。
- 数据库通过 `carrier_id + idempotency_key` 唯一约束兜底。
- 捕获唯一键冲突后查询旧记录，并返回 `duplicate=true`。

这个点适合回答“如何保证接口幂等？”

### 3. 为什么保留原始事件？

原始事件不能被覆盖，因为它是排查问题和重放状态的证据。

保留原始事件的价值：

- 审计：证明物流商到底推了什么。
- 排错：对照原始字段和标准化结果。
- 重放：映射规则调整后可以重新处理。
- 幂等：原始事件表承担第一层去重。

这个点适合回答“为什么不直接存处理后的状态？”

### 4. 为什么用适配器模式？

不同物流商格式不同：

- A 使用 `mailNo/status/eventTime`。
- B 使用 `tracking_number/event_code/occurred_at`。

如果业务 Service 里写大量 `if carrier == A else if carrier == B`，后续新增物流商会越来越乱。

TrackFlow 抽象了 `CarrierAdapter`，每个物流商负责自己的字段解析、时间解析和状态提取，主流程只处理统一事件。

这个点适合回答“怎么做扩展性设计？”

### 5. 主动对账解决什么问题？

Webhook 不是 100% 可靠，可能漏推。系统不能永远等，所以需要主动查询物流商轨迹，把缺失事件补回来。

TrackFlow 的做法是：

- 对异常或人工选择的运单发起对账。
- 对账结果以 `RECONCILIATION` 来源写入标准事件表。
- 补入事件后重新执行状态机重建。
- 重复对账不会重复插入同一补偿事件。

这个点适合回答“如何处理第三方系统不可靠？”

## 面试官可能追问

### Q1：为什么不用“状态枚举排序大小”判断最终状态？

因为物流状态不是简单线性大小关系。比如 `RETURNING` 和 `DELIVERED` 不是谁数字大谁正确；`DELIVERY_FAILED` 后还可以再次派送。必须通过业务允许的状态转移关系判断。

### Q2：如果两个事件 eventTime 一样怎么办？

用 `receivedTime` 做第二排序；如果还一样，用数据库主键 `id` 做稳定排序，保证每次重放结果一致。

### Q3：终态后又收到新事件怎么办？

终态包括 `DELIVERED`、`RETURNED`、`CANCELLED`。正常情况下不再允许转移。事件仍然保存，但 `applied_to_state=false`，并生成非法状态转移异常。

### Q4：为什么重复事件还要返回成功？

Webhook 发送方通常只关心“你是否收到”。如果重复事件返回 500，物流商会继续重试，反而放大流量。正确做法是返回成功，并明确 `duplicate=true`。

### Q5：项目哪里还能优化？

- 接入 RabbitMQ 消费者和死信队列。
- 使用 Testcontainers 做 MySQL/RabbitMQ 集成测试。
- 用 WireMock 模拟物流商 429、500、超时和非法 JSON。
- 前端增加 ECharts 图表和更完整的分页筛选。
- 把 JDBC 查询拆成 Repository 层，提升代码组织性。

## 一句话总结

这个项目的核心不是“做了一个后台”，而是围绕第三方物流事件不可靠的问题，设计了一套可验签、可幂等、可重放、可追溯、可补偿的事件治理流程。
